# Agent 가이드라인 (AGENT.md)

이 프로젝트에서 작업하는 AI Agent와 개발자는 아래의 규칙을 준수해야 합니다. 
프로젝트의 상세한 구조, 기술 스택, 세계관 및 추가 개발 계획 등은 반드시 [DEVELOPMENT.md](DEVELOPMENT.md) 문서를 참고해 주세요.

## 1. 포트 할당 규칙
브랜치를 나누어 병렬로 작업할 때 서버 포트 충돌을 방지하기 위해 브랜치 이름에 기반하여 테스트 포트를 할당합니다.
- **규칙**: 브랜치 이름에 포함된 번호를 활용하여 `3000 + 번호` 형태로 포트를 설정합니다.
- **예시**: 작업 중인 브랜치가 `10/agy`라면, 테스트 포트는 `3010`번을 사용합니다.
- **예시**: 브랜치 이름이 `12/feature`라면, 포트는 `3012`번을 사용합니다.
- 코드를 작성할 때 포트를 하드코딩하지 말고 환경 변수(예: `process.env.PORT`)를 사용해 동적으로 변경할 수 있게 처리해야 합니다.

## 2. 브랜치 작업 및 충돌 방지 지침
서로 다른 브랜치에서 동시에 작업하면서 발생하는 코드 충돌을 최소화하기 위해 다음 사항을 따릅니다.

1. **독립된 브랜치 생성**: 모든 새로운 작업은 기준 브랜치(예: `main` 또는 `develop`)에서 분기된 새로운 브랜치에서 진행합니다. (예: `git checkout -b 10/agy`)
2. **작업 범위 최소화**: 본인의 작업 목표에 해당하는 파일과 디렉토리만 수정합니다. 공통 컴포넌트나 글로벌 설정 파일을 수정해야 할 경우, 변경을 최소화하고 명확한 주석을 남깁니다.
3. **주기적인 동기화**: 작업 중간에 틈틈이 기준 브랜치의 최신 커밋을 가져와(`git pull --rebase origin main`) 충돌을 사전에 발견하고 해결합니다.
4. **작은 단위의 커밋**: 변경 사항을 논리적인 최소 단위로 쪼개어 자주 커밋합니다. 이는 충돌 발생 시 원인을 파악하고 롤백하기 쉽게 만듭니다.
5. **PR(Pull Request) 전 자체 리뷰**: 메인 브랜치로 병합하기 전, 변경된 파일을 꼼꼼히 확인하여 다른 브랜치 작업에 영향을 줄 수 있는 요소가 없는지 점검합니다.

## 3. 문서화 및 인수인계 지침
작업 중이거나 완료된 후, 사람이 코드를 쉽게 이해하고 유지보수 및 추가 개발을 원활하게 할 수 있도록 친절한 문서를 작성해야 합니다.

1. **상세한 주석 작성**: 복잡한 로직, 중요한 비즈니스 규칙, 또는 예외 처리가 포함된 코드에는 반드시 '왜(Why)' 그렇게 작성했는지와 '어떻게(How)' 동작하는지 설명하는 주석을 남깁니다.
2. **문서(README, 가이드) 업데이트**: 새로운 기능이 추가되거나 환경 설정이 변경될 경우, 프로젝트의 `README.md` 등 관련 가이드 문서를 즉시 최신화하여 누구나 변경 사항을 알 수 있도록 합니다.
3. **히스토리 및 의사결정 기록**: 작업을 진행하면서 내린 중요한 기술적 의사결정이나 이슈 해결 과정은 문서나 PR(Pull Request) 본문에 상세히 기록하여 추후 참고 자료로 활용합니다.
4. **가독성을 고려한 개발**: 코드 자체가 문서의 역할을 일부 수행할 수 있도록, 직관적인 변수/함수명과 명확한 구조로 작성합니다.
